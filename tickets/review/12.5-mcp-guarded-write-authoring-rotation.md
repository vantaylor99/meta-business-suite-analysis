## What landed

Added the two remaining guarded-write families to the custom Meta MCP server ‚Äî **authoring**
(create-only, PAUSED by default) and **audience rotation** ‚Äî as `propose_*` tools that reuse the
ticket-12 propose/execute machinery. No new gate, no raw Graph client: every tool wraps an existing
library builder and routes through the single `proposals.execute_plan(plan_id)` writer.

New MCP write tools (in `build_write_tools`, each `_wrap_tool_errors`-wrapped):
- Authoring: `propose_create_campaign`, `propose_create_adset`, `propose_create_ad`,
  `propose_create_video_ad`, `propose_duplicate_ad`, `propose_lookalike`.
- Rotation: `propose_audience_rotation`, `propose_advantage_disable`.

Write-tool count went 13 ‚Üí 21 (11 control + 6 authoring + 2 rotation + `preview_plan` + `execute_plan`).

### Key implementation facts to verify against

- **New library builders** (`authoring.py`): `build_create_campaign_plan`, `build_create_adset_plan`,
  `build_create_ad_plan` did **not** exist before (AGENTS.md listed `create_campaign`/`create_adset` as
  "hand-authored"). They were added, funneling through a shared `_build_netnew_create_plan` that reuses
  `validate_authoring_op` (required-param enforcement ‚Äî not duplicated), `_attach_netnew_grounding`
  (zero-sample ‚Üí `abstain`), and `_wrap_plan` (review + `requires_grounding`). `propose_create_video_ad`
  wraps the pre-existing `build_video_ad_plan`; `propose_duplicate_ad`/`propose_lookalike` wrap the
  pre-existing `build_duplicate_ad_plan`/`build_lookalike_plan`.
- **Dispatch** (`proposals.PLAN_APPLIERS`): added `authoring` (drops the `reader` kwarg ‚Äî authoring reads
  nothing at apply time), `audience_rotation`, `advantage_disable` (both keep `reader`). Each map key IS
  the `plan_type` its builder stamps.
- **Items-key split** (`proposals.PLAN_ITEMS_KEY` + `plan_items()`): rotation plans carry **no**
  `plan["ops"]` ‚Äî items live under `"rotations"`/`"items"`. The approval count (step 4), the proposal
  summary, `preview_plan`, and result serialization all now consult `plan_items()` rather than assuming
  `"ops"`. **Without this, a genuinely-approved rotation plan would be refused / crash.**
- **Result serialization** is plan-type-aware (`_serialize_results`): `OpResult`/`AuthoringResult` key by
  `op_id`; `RotationResult` keys by `adset_id` and has no `op_id`/`request` ‚Äî the old ops serializer
  would `AttributeError` on it.
- **Outcome verification** is plan-type-aware (`_verify_outcomes` dispatcher):
  - authoring ‚Üí re-reads each created entity's `effective_status`, red-flags a create that came back
    ACTIVE (a `created_active` follow-up), skips lookalike (inert audience, no status).
  - rotation ‚Üí re-reads each executed ad set and records the new audiences + `advantage_audience` flag.
  - ops ‚Üí unchanged (`verify_next_day_spend` follow-up).
- **Audit**: `_write_audit` dispatcher writes the family's timestamped results log; `plan_id` was added
  to `write_authoring_results` / `write_rotation_results` / `write_advantage_disable_results` for parity.
- **Docs**: `docs/META_ACTION_WORKFLOW.md` MCP section extended (authoring PAUSED-by-default + rotation
  + family-aware outcome verify); media-upload exception reiterated (`propose_create_video_ad` /
  `propose_create_ad` take an already-uploaded asset id ‚Äî the server never uploads).

## Known gaps / things to scrutinize (treat tests as a floor)

- **Net-new creates need an out-of-band `requires_grounding` drop to execute.** A net-new
  campaign/adset/ad cites a zero sample ‚Üí `abstain`, so an *approved* net-new create is hard-blocked at
  apply (the cold-create boundary). There is **no MCP tool/param to drop `requires_grounding`** ‚Äî the
  tests simulate the override by editing the persisted plan's guardrail. In practice today an approved
  net-new create can only execute via MCP if the persisted proposal is hand-edited, OR via
  `propose_duplicate_ad` (grounded on a proven source). This mirrors the CLI's documented behavior, but a
  reviewer should confirm the intended operator path (likely the `mcp-local-approval-gate` follow-on owns
  the override seam). `propose_lookalike` and `propose_advantage_disable` are structural abstains ‚Üí gate-
  allowed, so they execute on approval alone.
- **`preview_plan` for authoring/rotation reports stored intent, not a re-derived Graph request.** Ops
  plans re-derive via `control._build_request`; authoring/rotation items aren't ops-shaped (no `op["op"]`),
  so preview reports `params` / `diff` / new audience ids instead. Intentional (avoids a crash), but
  confirm it's the wanted preview fidelity.
- **AGENTS.md write-tool catalog is stale for the new proposers.** The rows for `create_campaign` /
  `create_adset` still read "hand-authored" and don't mention the MCP `propose_*` tools or the new
  `build_create_*_plan` library builders. Left untouched because AGENTS.md was not in this ticket's
  `files:` scope ‚Äî a reviewer may want a small doc-sync ticket.
- **Rotation outcome read-back is best-effort.** It re-reads targeting to confirm the write registered;
  in the unit tests the fake `update_adset` does not mutate state, so the read-back reflects pre-write
  targeting ‚Äî real Meta reflects the post-write state. The verify never raises and never gates the write.
- **`build_video_ad_plan` / `build_lookalike_plan` defer `validate_authoring_op` to apply time**, whereas
  the three new net-new builders validate at build (propose-time fail-fast). So a bad ratio / missing
  video_id on those two surfaces as an execute-time block rather than a propose-time `ToolError`. Both are
  clear; just an asymmetry worth noting.
