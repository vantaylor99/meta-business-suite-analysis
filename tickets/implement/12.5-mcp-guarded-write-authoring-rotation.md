## Why

`mcp-guarded-write-tools` builds the shared propose/execute machinery (proposal store, `execute_plan`
orchestration, approval seam, audit, outcome verify, token-scope error) and wires the `ops` plan
family. This ticket adds the two remaining guarded write families — **authoring** (create, PAUSED-by-
default) and **audience rotation** — as `propose_*` MCP tools that reuse that same machinery. No new
gate, no raw Graph client; every tool wraps an existing builder and routes through `execute_plan`.

## Design (resolved)

### Tool surface

Authoring propose tools — each wraps an `authoring.py` builder (which already attaches Evidence + a
computed confidence band, runs `review.review_authoring_plan`, and forces **PAUSED** on every spending
create in `authoring._build_create`, regardless of any review verdict):

- `propose_create_campaign`, `propose_create_adset`, `propose_create_ad`, `propose_create_video_ad`
  (validated by `authoring.validate_authoring_op`; required params enforced there — reuse, don't
  duplicate).
- `propose_duplicate_ad` (→ `build_duplicate_ad_plan`), `propose_lookalike` (→ `build_lookalike_plan`;
  audiences are inert — no PAUSED to force).
- **No delete / archive.**

Rotation propose tools — each wraps a `rotation.py` builder (these plans do **not** live under
`plan["ops"]`; each keeps its items under a plan-type key — `audience_rotation→"rotations"`,
`advantage_disable→"items"` — and is reviewed by `review.review_rotation_plan`):

- `propose_audience_rotation` (→ `build_rotation_plan`) — rotate exclusions forward by `offset`.
- `propose_advantage_disable` (→ `build_advantage_disable_plan`) — turn Advantage Audience off in
  place, audiences preserved.
- Bulk ad-set rename (`build_rename_plan`) is **out of scope** — the control `rename` op (ticket 12)
  already covers renames; note the deferral, don't build it.

All of these persist via `proposals.save_proposal` and return the same review-ready summary shape as the
control propose tools. Execution is the existing `execute_plan(plan_id)` — no new execute tool.

### Register the apply branches in the dispatch map

Extend `proposals.PLAN_APPLIERS` (left open in ticket 12) with the authoring + rotation branches:

```
"authoring":         lambda plan, client, **kw: authoring.apply_authoring_plan(
                         plan, client, **{k: v for k, v in kw.items() if k != "reader"}),  # no reader kw
"audience_rotation": lambda plan, client, **kw: rotation.apply_rotation_plan(plan, client, **kw),
"advantage_disable": lambda plan, client, **kw: rotation.apply_advantage_disable_plan(plan, client, **kw),
```

`apply_authoring_plan` has **no** `reader` parameter (authoring reads nothing live at apply time — it
POSTs creates); the wrapper drops the `reader` kwarg for it. `apply_rotation_plan` /
`apply_advantage_disable_plan` accept `reader` (live targeting drift re-read) and keep it. The `plan_type`
each builder stamps (`authoring`, `audience_rotation`, `advantage_disable`) is the dispatch key — verify
each builder's `plan_type` matches its map entry (a mismatch silently reviews/applies nothing).

### Outcome verification

Authoring: after execute, the store already records the created id per op. Add a read-back of each
created entity's `effective_status` via the `reader` to confirm it is **PAUSED** (the create-then-verify
lesson) — a create that comes back ACTIVE is a red flag surfaced in the result.

Rotation: `apply_rotation_plan` already does a live targeting drift re-read pre-write; the post-execute
outcome read from ticket 12's orchestration confirms the write registered.

### mcp_server + docs

- Extend `build_write_tools` with the authoring + rotation `propose_*` callables (same registration
  pattern, `_wrap_tool_errors`).
- Extend `docs/META_ACTION_WORKFLOW.md` with the authoring (PAUSED-by-default) + rotation entries and

  `create_video_ad`/`create_ad` with already-uploaded asset ids).

## Edge cases & interactions

- **PAUSED-by-default** — every executed authoring create comes back PAUSED; the outcome read asserts
  it. A lookalike audience has no status (not in `PAUSED_KINDS`) — don't assert PAUSED on it.
- **plan_type ↔ dispatch key drift** — a rotation plan routed through the `ops` iterator reviews
  nothing (the #1 rotation failure mode). Assert each builder's `plan_type` maps to the right applier.
- **Reader-kwarg split** — authoring apply must not receive `reader`; rotation apply must. Test both
  paths execute through `execute_plan` without a `TypeError`.
- **create_video_ad without an uploaded video_id** — `validate_authoring_op` already requires
  `video_id`/`page_id`/`link`; surface its `ValueError` as a clear tool error. Uploads are not an MCP
  tool (documented exception) — the id comes from a prior CLI upload.
- **Propose-then-execute with no approval** — same refusal as ticket 12 (0 approved ops); re-verify for
  an authoring and a rotation plan.
- **advantage_disable is OFF-only** — never re-enables Advantage Audience; already enforced in the
  builder — assert the direction can't invert.
