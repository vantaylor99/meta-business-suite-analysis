## What was built (implement stage)

Both `monitor.py` and `experiment.py` now emit the shared `confidence.py` `Confidence` band +
`Evidence` block the action plan already uses, so the whole repo speaks ONE confidence vocabulary.
The sample-size discipline that already lived in these modules (monitor's `min_spend` significance
floor + grace; experiment's `min_conversions` gate) was generalized into the shared `abstain`
verdict rather than reinvented.

- **monitor.py** — `classify_ad` gained `recency_days: int | None = 0` and returns a `confidence`
  dict on every branch: significance floor and protective grace → ⚪ abstain;
  `urgent`/`underperforming`/`ok` → `assess(...)` with `tier=direct_observation`. `build_watch_report`
  attaches `confidence` + a full reproducible `evidence` block (with `regenerating_query`) to each row.
- **experiment.py** — `read_experiment` adds `confidence` + `evidence` blocks alongside the unchanged
  `verdict`/`caveat` strings. Below `min_conversions` → abstain; `SIGNIFICANT` (p<0.05) → can reach
  🟢 high (A/B is the top grounding tier); `NO significant difference` (p≥0.05) → capped at medium.

### Findings and dispositions

- **MINOR — fixed inline. `Confidence` was being hand-constructed outside `confidence.py`.** The
  implementer flagged this himself ("worth a second opinion"): `monitor._abstain_confidence` built a
  `Confidence` object directly, eroding `confidence.py`'s stated invariant that the scoring logic
  lives in exactly one place and every band flows through that module. The rationale was sound (grace
  — "too young" — cannot be expressed as a below-floor *sample*, so `assess` has no input that drives
  it to abstain). Resolution: added a sanctioned **`confidence.abstain_confidence(...)` factory** that
  pins the data axis to `Band.abstain` (the integer floor, never a caller-supplied number) and reports
  the tier's honest grounding ceiling. `monitor._abstain_confidence` now delegates to it. This keeps
  every `Confidence` construction inside `confidence.py` while preserving the anti-fabrication
  invariant — `abstain` is the *absence* of a score, the one band reachable without sample inputs.
  Two tests pin the factory: it abstains on the data axis while keeping the grounding ceiling, and it
  exposes no caller-set band knob (mirroring `test_assess_exposes_no_pre_baked_band_parameter`).

- **MINOR — fixed inline. Untested branches.** The implementer's tests covered `insufficient`,
  grace-with-zero-sample, `urgent`, and all experiment paths, but not (a) grace when the sample is
  *large* — the exact case the abstain-vs-assess design rationale rests on — nor (b) the
  `underperforming` band. Added `test_classify_ad_young_ad_with_large_sample_still_abstains`
  (5k spend / 400 purchases, still ⚪ abstain — proves grace is not a sample-size verdict) and
  `test_classify_ad_underperforming_carries_direct_observation_confidence` (30 purchases → 🟡 medium,
  `direct_observation`, non-causal).

- **NOT A DEFECT — verified, left as-is. `urgent` ads usually read 🔴 low.** Confirmed intended: a
  flagged loser typically has thin conversions (< 25) even with cleared spend, so the data axis lands
  low. The monitor flags for the agent to reason over, not to auto-pause; reading low is honest. Pinned
  by `test_classify_ad_urgent_confidence_is_direct_observation_non_causal`.

- **NOT A DEFECT — verified. Monitor recency hard-coded to 0 / experiment `sample_spend=None` +
  `spend_floor=1.0`.** Both are correct for their callers: the watch window always ends at `as_of`
  (genuinely fresh), and the A/B's significance rests on conversions, not spend, so spend is kept off
  the data axis and `min_conversions` alone governs abstention. Documented in code comments. No change.

- **THEORETICAL ONLY — no code change. "enough conversions but p_value is None" → verdict says
  "could not compute significance" while confidence would not abstain.** Reachable only if total
  purchases ≥ total impressions (pooled ≥ 1) — impossible given purchases ≤ impressions, so the branch
  is dead in practice. Added no defensive code (it would be unreachable/dead). Noted here for the
  record.

- **DOCS — checked, none stale.** `docs/META_ACTION_WORKFLOW.md`'s "Evidence and Confidence" section
  describes the *action plan* output (the actions ticket), not the watch report or experiment readout,
  and does not claim those lack confidence — so it is not made stale. `knowledge/README.md`'s band
  rubric (pinned by a test) is the same vocabulary these modules emit — unchanged and still accurate.
  No dedicated schema doc exists for `watch_report.json` or the experiment readout JSON.

- **Read-only w.r.t. Meta** preserved — both modules only enrich output dicts; no new writes.
