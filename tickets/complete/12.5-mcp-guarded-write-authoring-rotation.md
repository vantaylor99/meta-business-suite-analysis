## What shipped (implement stage, commit 89c89f4)

Two guarded-write families added to the custom Meta MCP server as `propose_*` tools that reuse the
ticket-12 propose→validate→execute→verify machinery:

- **Authoring** (create-only, forced PAUSED): `propose_create_campaign`, `propose_create_adset`,
  `propose_create_ad`, `propose_create_video_ad`, `propose_duplicate_ad`, `propose_lookalike`.
- **Rotation**: `propose_audience_rotation`, `propose_advantage_disable`.

Write-tool surface went 13 → 21. `proposals.py` gained plan-type-aware dispatch (`PLAN_APPLIERS`,
`PLAN_ITEMS_KEY` + `plan_items()`, `_serialize_results`, `_verify_outcomes` splitting into
ops/authoring/rotation verifiers, `_write_audit`). No new Graph client, no new gate — every tool
routes through the single `execute_plan` writer.

## Review findings

**Read the full implement diff first (all six changed files), then the handoff.** The implementation
is solid, well-factored (the dispatch/serialize/verify splits are genuinely DRY over the four
families), and the test suite is thorough. Findings below.

### Checked and found correct (no action)
- **Safety-flow parity.** Every new tool builds a reviewed, persisted plan and returns only a
  `plan_id` summary — `execute_plan` remains the sole writer, still refuses a zero-approved plan
  (verified for authoring AND rotation, whose items live under `rotations`/`items`, not `ops`), still
  runs the mandatory validate pass before any execute pass. Confirmed against `proposals.execute_plan`.
- **PAUSED-by-default.** `authoring._build_create` hardcodes `status="PAUSED"` on every spending
  create (campaign/adset/ad/video_ad); the create-then-verify re-read red-flags any created entity that
  comes back ACTIVE. Lookalike is correctly exempt (an audience is inert — no status to force/read).
- **Dispatch correctness.** Each builder's stamped `plan_type` is present in both `PLAN_APPLIERS` and
  `PLAN_ITEMS_KEY`, and its items are non-empty under the mapped key (guards the #1 rotation failure
  mode — a mismatch would silently approve/apply nothing). Reader-kwarg split verified: authoring applier
  drops `reader` (no `TypeError`), rotation/ops keep it. `advantage_disable` writes only `=0`,
  preserves audiences verbatim, and `compute_new_targeting` has no path that sets `1`.
- **Error handling / result-shape safety.** Outcome verify and result serialization are plan-type-aware;
  the ops serializer would `AttributeError` on a `RotationResult`, and the code correctly routes each
  result type by `plan_type`. All verify re-reads catch `MetaApiError` and record (never raise).
- **Reused-function signatures.** `build_rotation_plan`, `fetch_entity_metrics`, `_resolve_grounding_window`,
  `control._get_entity`, `reader.list_adsets(effective_status=...)`, `scope_error_from_results`
  (uses `getattr`, works across all result types) all match their call sites.

### Minor — fixed inline this pass
- **preview_plan for authoring/rotation was untested.** The implementer flagged that non-ops previews
  report stored intent (params / diff + new audiences) instead of a re-derived Graph request, but no
  test covered it. Added `test_mcp_preview_plan_authoring_and_rotation_report_stored_intent`
  (write-free, asserts the normalized `op`/`op_id` fields and the stored-intent `would_send` for both a
  net-new campaign create and a rotation). Confirms the documented non-ops path neither crashes nor
  writes. Suite: 429 → 430 passing.
- **AGENTS.md write-tool catalog was stale.** Rows for `create_campaign`/`create_adset`/`create_ad`
  still read "hand-authored" / named only `propose-duplicate-ad`. Updated both rows to name the new
  `build_create_*_plan` builders and the MCP `propose_create_*` tools (no CLI proposer ships), and to
  note the net-new grounding-override gap → see below.

### Major — filed as a new ticket (not fixed here)
- **Net-new creates can propose but not execute over MCP.** A net-new campaign/ad set/ad cites a zero
  sample → `abstain`, so an *approved* net-new create is hard-blocked at apply by the grounding gate —
  correct (the cold-create boundary), but there is **no MCP-side path to consciously drop
  `requires_grounding`** (the tests simulate it by hand-editing the persisted plan). So four of the six
  authoring tools are effectively propose-only over MCP today. This is an intentional block needing an
  operator-facing design decision, closely tied to the `mcp-local-approval-gate` human-confirmation
  seam. Filed `tickets/backlog/mcp-netnew-create-grounding-override.md` (prereq:
  `mcp-local-approval-gate`) with three resolution options. `propose_duplicate_ad` (grounded) and
  `propose_lookalike` (structural abstain) are unaffected — they execute on approval alone.

### Noted, no action (documented design, not defects)
- **preview fidelity for non-ops families** reports stored intent, not a re-derived request — intentional
  (authoring/rotation items are not ops-shaped; re-deriving would crash). Now covered by the new test.
- **Rotation outcome read-back is best-effort** — it re-reads targeting to confirm the write registered,
  never raises, never gates the write. Correct posture.
- **`build_video_ad_plan`/`build_lookalike_plan` defer `validate_authoring_op` to apply time** while the
  three new net-new builders validate at build — an asymmetry (execute-time block vs propose-time
  `ToolError`), both clear, not worth changing.
- **`adset_rename` plan_type** (bulk rename, intentionally not an MCP tool) is not in `PLAN_APPLIERS`;
  if ever routed through `execute_plan` it degrades to a graceful "no approved ops" refusal (defaults to
  the `ops` items key), not a crash. Pre-existing scope, not a regression.
